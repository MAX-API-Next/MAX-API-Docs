# 主流大模型信息跟踪服务建设方案

> 状态：方案草案  
> 日期：2026-07-20  
> 适用项目：MAX API、MAX-API-Docs、AIOAGI 文档站

## 1. 结论

建议新建一个独立项目，暂定名为 `Model-Intelligence-Hub`，负责主流大模型的官方动态、模型能力、价格、API 调用方式和废弃计划等信息的采集、标准化、审核与发布。

不建议把采集器和数据库直接放进 `MAX-API-Docs`，也不建议直接放进 AIOAGI 的 Mintlify `docs` 项目。两个项目都应作为展示端，通过构建脚本或版本化 JSON 快照消费已经审核的数据。

推荐关系如下：

```text
官方文档、价格页、官方博客、官方 SDK、官方模型接口
                         |
                         v
              Model-Intelligence-Hub
       采集 -> 原始快照 -> 标准化 -> 差异检测 -> 人工审核
                         |
          +--------------+---------------+
          |              |               |
          v              v               v
  MAX API 候选配置   MAX-API-Docs    AIOAGI docs
  测试与开发告警     产品技术文档     平台模型与价格说明
```

这套服务可以明显改进 MAX API 的开发工作，但必须定位为“外部事实与变更建议来源”，不能成为直接执行生产扣费的第二套计费系统。

## 2. 为什么应该独立建项目

### 2.1 三种方案对比

| 方案                  | 优点                                                   | 主要问题                                                                       | 结论               |
| --------------------- | ------------------------------------------------------ | ------------------------------------------------------------------------------ | ------------------ |
| 集成进 `MAX-API-Docs` | 页面开发和展示方便                                     | 文档构建与定时采集、数据库、审核任务生命周期不同；会增加构建失败和密钥暴露风险 | 不推荐作为核心服务 |
| 集成进 AIOAGI `docs`  | 可以快速更新平台文档                                   | Mintlify 项目适合静态内容，不适合运行采集器；平台零售价容易与厂商参考价混淆    | 仅作为消费端       |
| 新建独立服务          | 一套数据服务多个项目复用；可以保留历史、证据和审核记录 | 初期需要多维护一个部署单元                                                     | 推荐               |

### 2.2 职责边界

- `Model-Intelligence-Hub`：维护外部模型事实、来源证据、历史版本和变化事件。
- `MAX API`：维护实际支持的渠道、模型映射、用量归一化和生产计费规则。
- `MAX-API-Docs`：解释 MAX API 的能力、兼容方式、配置方法和已支持特性。
- AIOAGI `docs`：解释 AIOAGI 实际开放的模型、平台售价、分组和使用方法。

独立服务不应保存 MAX API 或 AIOAGI 的用户 API Key，也不应直接连接生产数据库。

## 3. 当前项目基础

MAX API 已经具备一部分可复用能力，不需要从零重新实现：

1. `controller/model_sync.go` 已支持从 `basellm/llm-metadata` 风格接口同步模型和供应商元数据，并提供 `SYNC_UPSTREAM_BASE` 配置入口。
2. `controller/ratio_sync.go` 已支持 `/api/pricing`、`/api/ratio_config` 和 `models.dev` 价格数据格式。
3. `pkg/billingexpr/expr.md` 已定义分阶段计费表达式，价格系数可以表达每百万 Token 的真实价格以及缓存、图片、音频、长上下文等条件。
4. `BillingSession` 和现有计费表达式应继续作为 MAX API 的计费执行真相来源。
5. AIOAGI 文档站已经有 `models.mdx` 和 `pricing.mdx`，适合作为审核后信息的发布入口。

因此，新服务的重点不是再造模型列表或计费引擎，而是补齐多来源采集、证据留存、差异检测、审核发布和开发反馈闭环。

## 4. 必须分开的四类价格

以下数据不能存成同一个“模型价格”字段：

| 价格类型             | 所有者                 | 用途                         | 是否允许自动覆盖         |
| -------------------- | ---------------------- | ---------------------------- | ------------------------ |
| 厂商官方参考价       | Model-Intelligence-Hub | 外部事实、成本分析、变更提醒 | 可自动采集，不可直接生效 |
| MAX API 默认计费建议 | MAX API 代码或发布快照 | 为管理员提供初始配置         | 只能经审核和测试后更新   |
| MAX API 部署实例价格 | 部署管理员             | 实际生产扣费                 | 禁止外部服务自动覆盖     |
| AIOAGI 平台销售价格  | AIOAGI 控制台          | 面向最终用户计费和展示       | 以控制台为准，独立审批   |

厂商降价不代表平台应立即降价，厂商涨价也不能未经核对就直接修改线上扣费。任何价格同步都必须先生成差异，再由管理员决定是否应用。

## 5. 跟踪范围

### 5.1 模型基础信息

- 厂商、模型系列、稳定模型 ID 和别名。
- 发布、预览、正式可用、废弃、停止服务等生命周期状态。
- 上下文长度、最大输出长度和知识截止时间。
- 文本、图像、音频、视频、Embedding、Rerank、工具调用和结构化输出能力。
- 区域、账号等级、白名单或服务层级限制。

### 5.2 API 调用方式

- 官方 Base URL、认证方式和端点。
- Chat Completions、Responses、Messages、原生 Gemini 等协议支持情况。
- 流式响应、批处理、实时接口、文件接口和异步任务接口。
- 请求参数、响应结构、usage 字段和错误码变化。
- 官方 SDK 最低版本和调用示例。

示例代码应保存为经过校验的模板，不应在数据库中保存真实 API Key。

### 5.3 价格信息

- 币种、地区、税前或税后口径。
- 输入、输出、缓存读取、缓存创建、音频、图片和视频价格。
- 每 Token、每百万 Token、每秒、每张、每次任务等计价单位。
- Batch、Priority、Flex、长上下文和不同时段等价格条件。
- 生效时间、失效时间、来源页面和最后确认时间。

所有金额使用定点小数或十进制字符串保存，禁止使用二进制浮点数作为价格存储格式。

### 5.4 动态与资讯

- 官方发布日志、官方博客和官方 GitHub Release。
- 模型上线、改名、能力变化、价格变化和废弃通知。
- 官方 SDK、OpenAPI Schema 和 usage 字段变化。
- 第三方媒体内容只作为发现线索，不能直接成为价格或 API 事实来源。

首期不做社交媒体舆情、模型排行榜和自动生成的主观评测。

## 6. 数据来源与可信度

### 6.1 来源等级

| 等级 | 来源                                                | 可用于什么                          |
| ---- | --------------------------------------------------- | ----------------------------------- |
| P0   | 厂商官方文档、官方价格页、官方模型 API              | 价格、模型 ID、端点和限制的主要依据 |
| P1   | 官方博客、官方 GitHub、官方 SDK 和 Release          | 发布事件、协议和字段变化            |
| P2   | `models.dev`、`basellm/llm-metadata` 等结构化聚合源 | 交叉校验和快速发现                  |
| P3   | 技术媒体、社区帖子和社交平台                        | 仅生成待核实线索                    |

### 6.2 发布规则

- 价格变化必须有 P0 来源，并经过人工审核后才能进入“已确认”快照。
- 新模型可以由 P0 或 P1 来源确认。
- P2 数据与 P0 冲突时，以 P0 为准并生成冲突告警。
- P3 信息只能标记为 `lead`，不能出现在正式价格导出中。
- 每条公开记录必须带 `source_url`、`observed_at`、`valid_from`、`source_hash` 和审核状态。

## 7. 核心数据模型

建议至少包含以下实体：

| 实体                    | 关键内容                                        |
| ----------------------- | ----------------------------------------------- |
| `providers`             | 厂商标识、名称、官方网站和状态                  |
| `models`                | 稳定内部 ID、官方模型 ID、系列、生命周期        |
| `model_aliases`         | 别名、日期别名、渠道映射和有效期                |
| `capability_profiles`   | 上下文、输入输出模态、工具和结构化输出能力      |
| `api_profiles`          | Base URL、协议、端点、认证和 SDK 信息           |
| `price_sheets`          | 厂商、币种、地区、版本和生效时间                |
| `price_tiers`           | 计价维度、单位、条件和十进制金额                |
| `source_records`        | 来源 URL、抓取时间、哈希、HTTP 元数据和许可备注 |
| `observations`          | 每次解析得到的候选事实                          |
| `change_events`         | 新增、修改、废弃、冲突和恢复事件                |
| `review_records`        | 审核人、结论、备注和时间                        |
| `publication_snapshots` | 对外发布的不可变版本、校验和和生成时间          |

模型 ID、模型别名和 MAX API 内部映射必须分开保存，避免把渠道别名误当成厂商正式模型名。

## 8. 采集与发布流程

### 8.1 标准流水线

1. 在来源注册表中人工登记允许访问的域名、URL、解析器和更新频率。
2. 使用 ETag、Last-Modified、超时、限速和重试获取原始内容。
3. 保存原始响应哈希和必要的审计快照。
4. 优先使用官方 JSON、OpenAPI、RSS 或 API；HTML 使用结构化解析器。
5. 将不同来源转换为统一候选数据，执行类型、单位和范围校验。
6. 与上一已发布快照比较，生成字段级差异。
7. 根据来源等级、变化幅度和冲突情况进入自动通过或人工审核队列。
8. 审核通过后生成不可变版本快照和各消费端导出文件。
9. 文档站和 MAX API 仅消费指定版本，不读取尚未审核的实时采集结果。

### 8.2 AI 的使用边界

AI 可以用于：

- 从非结构化发布说明中提取候选变更。
- 对同一变更生成中、英、日摘要草稿。
- 为人工审核生成差异说明。

AI 不可以用于：

- 在没有官方证据时推断价格。
- 自动决定生产计费配置。
- 跳过 Schema、单位和范围校验。
- 将网页中的指令作为系统操作执行。

## 9. 对外接口与导出

建议提供只读版本化接口：

```text
GET /v1/providers
GET /v1/models
GET /v1/models/{provider}/{model}
GET /v1/prices
GET /v1/events
GET /v1/changes?since=<snapshot>
GET /v1/snapshots/{version}
```

同时生成面向现有项目的兼容导出：

```text
/exports/max-api/models.json
/exports/max-api/vendors.json
/exports/max-api/pricing-candidates.json
/exports/docs/model-catalog.json
/exports/docs/model-events.json
```

每个导出文件应包含：

- `schema_version`
- `snapshot_version`
- `generated_at`
- `source_revision`
- `checksum`
- `data`

生产消费端应固定 `snapshot_version` 或校验 `checksum`，避免采集过程中的半成品进入构建或运行环境。

## 10. 与 MAX API 的集成方案

### 10.1 模型元数据

服务可以输出与现有 `basellm/llm-metadata` 相容的 `models.json` 和 `vendors.json`。测试环境可通过 `SYNC_UPSTREAM_BASE` 指向该服务，复用 MAX API 已有同步逻辑。

首期只建议：

- 创建缺失模型候选。
- 提示模型描述、端点和能力变化。
- 对已有本地模型生成差异，不自动覆盖。

### 10.2 价格候选

服务可以输出 MAX API 已支持的 `/api/pricing` 兼容结构，但默认只进入“比较和预览”流程：

1. 拉取厂商参考价和当前 MAX API 配置。
2. 展示按模型、计价维度和生效时间划分的差异。
3. 管理员选择需要更新的模型。
4. 生成 `tiered_expr`、倍率或固定价格候选。
5. 编译表达式并运行价格样例和结算测试。
6. 导出备份和待应用配置。
7. 经人工确认后才写入测试环境，验证后再进入生产环境。

禁止由定时任务直接调用 MAX API 管理接口修改线上价格。

### 10.3 开发反馈闭环

该服务可以把外部变化转换为 MAX API 的可执行开发任务：

- 新模型事件：生成渠道支持检查和模型元数据 PR 草稿。
- 新端点或协议：生成适配器差距报告。
- 新 usage 字段：生成用量归一化、日志和结算路径检查项。
- 价格维度变化：生成 `tiered_expr` 候选和测试向量。
- 模型废弃：扫描默认模型、文档、测试和示例中的引用。
- SDK 或 Schema 变化：生成请求和响应 Fixture，触发适配器回归测试。
- 上下文或能力变化：更新管理端模型说明和文档候选。

对于计费相关变化，测试必须覆盖客户端 usage、内部 `dto.Usage`、`BillingUsage`、最终额度和日志信息，不能只验证价格表本身。

## 11. 与两个文档站的集成

### 11.1 MAX-API-Docs

适合展示：

- MAX API 已支持模型和协议的能力矩阵。
- 新模型的接入方法和渠道配置说明。
- API 调用示例、兼容限制和废弃提醒。
- MAX API 版本与厂商变化的关联说明。

建议在构建前拉取一个已审核快照并生成 MDX。构建失败时继续使用仓库内上一次成功快照，避免外部服务故障阻断文档发布。

### 11.2 AIOAGI docs

适合展示：

- AIOAGI 当前实际开放的模型列表。
- 平台分组、可用性、调用方法和平台价格。
- 厂商动态摘要及其对 AIOAGI 用户的影响。

AIOAGI 页面必须将“厂商官方参考价”和“AIOAGI 实际售价”明确分栏。实际可用模型应由 AIOAGI 的 `/models` 或内部审核导出决定，不能仅因为外部服务发现了模型就宣称平台已经支持。

Mintlify 项目只接收生成后的 MDX 或 JSON 快照，不在文档构建过程中运行网页采集器。

## 12. 推荐技术方案

### 12.1 首期技术栈

- 运行时：Bun + TypeScript。
- API：Hono 或同等级轻量 HTTP 框架。
- 校验：Zod 和 JSON Schema。
- 数据库：PostgreSQL。
- 数据访问：Drizzle ORM 或项目统一的 SQL 层。
- HTML 解析：Cheerio；只有必须执行 JavaScript 的官方页面才使用 Playwright。
- 调度：首期使用单实例 Cron 和数据库租约，规模扩大后再引入队列。
- 原始快照：数据库元数据加私有对象存储；MVP 可先使用压缩文件存储。
- 部署：独立 Docker Compose 服务。

首期不要把 Next.js 作为采集服务核心，也不要为了少量定时任务立即引入 Redis、Kafka 或 Kubernetes。

### 12.2 建议目录

```text
Model-Intelligence-Hub/
├─ apps/
│  ├─ api/                  # 只读查询和审核接口
│  └─ worker/               # 定时采集、解析和差异检测
├─ packages/
│  ├─ schema/               # Zod、JSON Schema 和数据类型
│  ├─ collectors/           # 按厂商拆分的采集适配器
│  ├─ normalizers/          # 模型、价格、API 和事件标准化
│  ├─ diff/                 # 字段级比较和风险分级
│  └─ exporters/            # MAX API 和文档站导出器
├─ sources/                 # 来源注册表和域名允许列表
├─ fixtures/                # 去敏后的官方响应样例
├─ migrations/              # 数据库迁移
├─ docs/                    # 服务自身的运维和数据规范
└─ docker-compose.yml
```

## 13. 安全、合规和可靠性要求

- 只允许访问来源注册表中的 HTTPS 域名，防止 SSRF。
- 不抓取需要绕过登录、验证码、付费墙或访问控制的内容。
- 遵守站点条款、robots 规则、许可和合理请求频率。
- 对外只发布结构化事实、短摘要和来源链接，不复制整篇受版权保护的文章。
- 原始页面快照默认私有，仅用于审计和差异确认。
- 日志不得记录 API Key、Cookie、Authorization Header 或管理接口凭证。
- 所有抓取内容按不可信输入处理，禁止执行其中的脚本或操作指令。
- 价格异常变化、单位变化和大范围模型删除必须强制人工审核。
- 采集失败不得把旧值改成空值，也不得发布“全量删除”快照。
- 每次发布支持回滚到上一快照。

## 14. 分阶段实施计划

### 阶段 0：数据契约和来源清单（3 至 5 天）

- 明确四类价格和各项目的数据所有权。
- 定义模型、能力、API、价格、来源和事件 Schema。
- 建立首批官方来源注册表和域名允许列表。
- 定义 MAX API 与两个文档站的导出契约。

验收标准：可以用固定 Fixture 生成一个带来源、时间和校验和的版本快照。

### 阶段 1：最小可用采集服务（1 至 2 周）

- 接入 3 至 4 个最重要厂商的官方结构化来源。
- 支持原始快照、标准化、字段级 Diff 和来源健康检查。
- 支持价格变化强制人工审核。
- 提供只读 API 和 Docker Compose 部署。

验收标准：连续运行 7 天，没有重复事件、空值覆盖和无法追溯来源的数据。

### 阶段 2：文档站消费（1 周）

- 为 `MAX-API-Docs` 生成能力矩阵和模型动态候选 MDX。
- 为 AIOAGI `docs` 生成已开放模型和厂商动态候选内容。
- 增加最后核验时间、来源链接和快照版本。
- 外部服务不可用时回退到上一次快照。

验收标准：两个文档站都能使用固定快照完成离线构建，且不会混淆厂商参考价和平台售价。

### 阶段 3：MAX API 开发集成（1 至 2 周）

- 输出现有模型同步格式和 `/api/pricing` 候选格式。
- 自动生成新模型、协议变化、usage 变化和废弃事件报告。
- 为计费表达式生成测试向量，不自动应用生产价格。
- 在 CI 中运行模型 Schema、调用 Fixture 和计费回归测试。

验收标准：一次外部模型或价格变化能够形成“证据 -> 差异 -> 测试 -> 审核 -> PR/配置候选”的完整链路。

### 阶段 4：生产强化（持续）

- 增加多实例任务租约、告警、监控和备份。
- 扩展更多厂商、区域和多模态计费方式。
- 增加来源冲突、长期未更新和解析器失效监控。
- 基于实际价值决定是否增加审核 Web UI。

## 15. 建议指标

| 指标                               | 初始目标    |
| ---------------------------------- | ----------- |

## 17. 最终建议

这个功能值得做，并且能够持续改进 MAX API 的模型适配、计费准确性、文档时效和版本规划。最有价值的部分不是再做一个资讯列表，而是建立可追溯的结构化事实和变更事件，并把变化转成 MAX API 的测试、Issue、PR 和配置候选。

实施时应坚持以下原则：

- 数据服务独立，文档站只负责展示。
- 外部事实、MAX API 默认值、部署实例价格和 AIOAGI 售价严格分离。
- 自动采集和自动生成候选，人工决定生产生效。
- 每个结论有来源、时间、历史和回滚能力。
- MAX API 的 `BillingSession` 与计费表达式继续保持唯一生产计费真相。
